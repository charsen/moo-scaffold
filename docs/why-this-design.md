# 为什么这样设计

> 这篇给**想借鉴的人**看:不讲怎么用(用法看 [README](../README.md) 和 [docs/guide/](guide/README.md)),只讲**为什么这么写、什么时候该删**。如果你也在做个人 / 小团队的内部工具,这里的判断比代码本身更值钱。

moo-scaffold 是一个人多年 Laravel 经验的沉淀。它不追求"功能多",追求"每一块都简单到几乎不用维护"。支撑这个取舍的是下面几条尺子,和一串"建了又砍"的真实记录——**后者才是重点**。

## 一、五把尺子

读任何新功能 / PR 前,先拿这几把尺子量一遍:

**1. 功能必须简单,哪怕业务面很广。**
覆盖面不窄(代码生成、DB 文档、API 文档 + 调试、ACL、配置 UI、文档中心……),但每一块都**不堆复杂特性、不做多层兜底**。"备份历史 / 修改前快照 / 多步撤销 / 操作审计"这类——**默认劝退**。理由很简单:这些是 git 已经做得更好的事。能走 git 就走 git。

**2. 环境边界是定位的一部分,不是事后补的安全补丁。**
所有"写"操作(改配置、增删账号、生成代码、迁移 schema)**只在开发环境启用**,生产一律只读。两条强制路径:

- CLI:`config('scaffold.only_in_local')` 默认 `true`,非 local 环境所有生成命令直接退出。
- Web:`EnforceScaffoldWritable` 中间件在 `APP_ENV=production` 或 `SCAFFOLD_CONFIG_READONLY=true` 时拒绝所有 POST/PUT/PATCH/DELETE,读类页面照常。

关键推论:**因为有这条硬边界,整个工具不需要复杂权限模型 / 操作审计 / 撤销链。** 一条架构边界,省掉一整类功能——这是"约束即简化"最划算的一笔。

**3. UI/UE 可以追求完美,但完美 ≠ 功能复杂。**
设计 token、组件抽象、暗色适配、CSP 兼容、动效——这些值得反复打磨,不必质疑投入产出。但视觉投入**不能借口要求新增功能复杂度**。把"做得好看"和"做得复杂"分开。

**4. 代码生成模板就是编码规范。**
`stubs/*.stub` 和 `src/Foundation/` 的基类不是普通模板,它们是"一个 PHP 项目应该长成什么样"的规范本体。改这里要按"改团队规范"的严肃程度对待,而不是"随手改个模板解决眼前的单点问题"。**想借鉴这套代码风格,直接读 `stubs/` 就够了。**

**5. 这不是给陌生团队的通用工具。**
个人 / 小团队工具,**容错和环境适应性不是优先目标**。一处"用户可能没装这个、没配那个"的兜底,往往比"假设环境符合约定"的简洁版更糟——因为兜底会长出分支、长出配置项、长出文档,最后比它要解决的问题还重。

> 作者明确授权:当有人(包括作者自己)提出违反上述尺子的需求(典型:加快照 / 审计 / 多步撤销 / 给生产开写入口),**先提醒、再确认**,而不是直接照做。push back 是被授权的角色职责,不是顶撞。

## 二、砍功能史(本文件的重点)

判断力不体现在"做了什么",体现在"敢删什么"。下面每一条都是**建好、上线、甚至修过 bug,最后还是整层删掉**的真实记录(版本号对照 [CHANGELOG](../CHANGELOG.md)):

**接口调试「完成签收」整层下线(3.10.0)。**
曾做过一套团队共享的"谁调过哪个接口"签收系统:YAML 存储 + flock 锁 + 多用户合并 + 统计页 + 侧栏状态圈。还为它修过一个不小的 bug(取消签收会误删别人的记录)。上线后**实测:无人标注、无人看**。于是整层删除——路由、页面、按钮、状态圈、controller 方法、SCSS 全清。
→ **修过 bug 不是保留的理由,沉没成本不是资产。** 一个没人用的功能,留着就是纯负债。

**账号 API token 装置整体移除(3.4.2)。**
当年给 Chrome 扩展的 todo 上报做认证用。todo 改走云端后,这套 token 失去所有活的消费方(`findByToken` 全代码零调用)。于是整体拔除:存储字段、自动签发、UI 列、配置项、CLI flag、stub 字段。
→ **依赖方消失了,功能本身就该跟着消失。** 别留"以后可能用得上"的死装置。

**本地监控查看器退役 → 云端化(3.4.x → 3.9.0)。**
运行时异常 / 慢 SQL 一度在本地有完整查看器 + 一堆管理方法(prune / purge / resolve / reopen)。查看统一搬到云端后,本地这套全删;到 3.9.0,把整条"采集 → 缓冲 → 推送 → MCP"链路抽成独立包。
→ **同一件事不要有两个家。** 数据的真源只能有一个,另一个不管多顺手都是混乱源。

**配置中心 / 开发人员管理,上线一天内连砍 4 刀。**
功能上线后,在一天里被连续砍掉 4 个方向。不是"做错了",是"做多了"——上线后才看清哪些是真需求、哪些是自己想象出来的。
→ **上线是认识需求的开始,不是结束。** 砍掉自己刚做的东西,需要的不是技术,是诚实。

**配置 / env 键移除,不做兼容回落(3.7.x / 3.9.0)。**
删配置键、改 env 名,**不留向后兼容的回落分支**——检测到旧 env 残留就直接提示改名。
→ **对个人 / 小团队工具,一条清晰的路 > 两条都能走的路。** 兼容层是给陌生用户的,而尺子 #5 说了:这不是给陌生团队的。

**`moo:doctor` 加了又砍(同一周内)。**
加过一个"代码生成新鲜度体检"命令,几天后就砍了——因为生成器本身已经堵住了"用旧缓存"的口子,这个体检纯属累赘。
→ **新加的功能也照样上尺子,自己刚写的不享受豁免。**

## 三、几个值得直接借鉴的范式

只想拿走几样东西的话,看这几处:

**Schema → 缓存 → 生成器 的单一事实源流水线。**
用户写 `scaffold/database/*.yaml`;`moo:fresh` 解析成 `storage/scaffold/` 下的缓存;**所有生成器读缓存,不读 YAML**。YAML 给人看、缓存给机器用,两者解耦;一处解析、多处消费,不会有 N 个生成器各自解析 YAML 各自跑偏。
看 `src/Generator/` + `src/Command/FreshStorageCommand.php`。

**`stubs/` 即编码规范。**
不想读完整套代码,只想知道"这人觉得 controller / model / request 该怎么写"?直接读 `stubs/*.stub` 和 `src/Foundation/` 的基类。它们是规范的本体,不是注释里的口号。

**dev-only 写 / prod 只读 的双层强制。**
CLI 一层(`only_in_local`)、Web 一层(`EnforceScaffoldWritable` 中间件)。任何新写操作必须同时过这两关。看 `src/Http/Middleware/EnforceScaffoldWritable.php`——**一个中间件顶掉一整套权限系统**。

**路径穿越的双层防御。**
凡是用户可控的路径段(文档 slug、存储桶名)拼进文件读写的,**路由层 `->where()` 正则约束 + 存储层入口 `preg_match` 再校验**,两道都设。理由:路由哪天被改成泛匹配,存储层是最后兜底。看 `src/Support/DocsRepository.php` 的 slug 校验。

**CSP-safe 的前端写法。**
后台跑在 CSP 严格模式(Alpine 的 CSP build),模板里不允许内联表达式——所有逻辑写进注册好的方法。这套约束逼出的代码反而更干净(逻辑不散在模板里)。看 `public/javascript/alpine-init.js` 和 [docs/internal/framework-traps.md](internal/framework-traps.md)。

## 四、它不适合谁(说在前面)

诚实比好看重要:

- **想要开箱即用、自动适配你环境的通用脚手架** —— 不是它。它假设环境符合约定(目录结构、路由标记、宏注册),不为"你没配那个"做兜底。
- **想要纯图形化点点点生成、不碰 YAML / 命令行** —— 不是它。核心是"写 YAML + 跑命令",可视化设计器是辅助。
- **想要它管生产环境的数据 / 配置** —— 不是它。生产一律只读,这是设计意图。

它**适合**:愿意接受一套有主见的约定,换来"写 20 行 YAML 省掉 200 行手写样板",并且想顺手把作者沉淀的判断也借走的人。

---

借鉴这套东西,最值钱的不是某个生成器,是上面「砍功能史」里那种**对自己作品下手的能力**。代码谁都会写,删自己刚写的代码才难。
