# 贡献指南

先说定位,免得来回:**moo-scaffold 是一个有主见的「参考实现」,不是面向通用场景、开箱即用的脚手架。** 它沉淀的是一个人多年 Laravel 的取舍。这决定了贡献的口味。

## 欢迎

- **借鉴** —— `stubs/` / `src/Foundation/` 基类 / 设计判断,随便拿(MIT)。这本来就是开源它的主要目的。
- **Bug 报告 / 修复** —— 踩到确切的 bug(带复现)最受欢迎。
- **文档勘误** —— 错别字、过时说明、失效链接,小 PR 直接来。
- **贴合定位的小改进** —— 在现有约定内把某块做得更稳 / 更简。

## 大概率会被婉拒(先开 issue 聊,别直接写)

下面这些方向**默认劝退**,提 PR 前先聊:

- 为"通用 / 适配各种环境"加配置项、加兼容分支、加兜底 —— 它不是给陌生团队的通用工具。
- "备份历史 / 修改前快照 / 多步撤销 / 操作审计"这类 —— 能走 git 就走 git。
- 给生产环境开写入口 —— dev-only 写、prod 只读是设计意图,不是疏漏。
- 显著增加功能复杂度的特性,哪怕"看起来有用"。

不是不欢迎想法,是这工具的价值恰恰在"克制"。拿不准,开个 issue 说一句即可。

## 动手前

- **改 SCSS** → commit 前必跑 `npm run build:css`(否则下游拉不到新 CSS)。
- **改 `stubs/*.stub` 或 `src/Foundation/`** → 这是"编码规范本体",按改规范的严肃度对待,别随手改模板解决单点问题。
- **跑测试** → `composer test`(零 env 即全绿,fixture 自带);涉及前端资源时再跑对应的 `npm` 构建或 e2e 检查。

## Commit / PR

- commit 风格看 `git log`:`feat(scaffold): …` / `fix(scaffold): …` / `docs: …`,中文 subject 是常态。
- PR 写清"解决什么 + 为什么这么改",附复现 / 验证方式。

谢谢 —— 尤其谢谢那些"读懂了再借鉴"的人。
